面向二次开发的整合文档。涵盖架构、目录、数据模型、API、判题机制、前端结构、配置部署，以及常见扩展点的改法。

## 1. 项目概览

教学型在线评测系统（Online Judge），支持题库、判题、竞赛、班级作业、讨论、题解审核、管理后台等。

**技术栈**

- 后端：Python / Django 5 + Django REST Framework；JWT 认证（SimpleJWT）；Celery + Redis 异步判题；PostgreSQL；判题沙箱 LocalSandbox（开发）/ go-judge（生产）。
- 前端：React 18 + Vite + Ant Design 5；Monaco 编辑器；react-markdown + KaTeX；axios；react-router v6；dayjs。
- 部署：Docker Compose（postgres / redis / go-judge / backend / worker / frontend(nginx)）。
- 设计风格：纸面学术风（暖纸底 `#f4f3ee`、墨色 `#20281f`、松绿主色 `--pine #0d6e56`、点缀金 `--gold #c98a12`），思源宋体标题 + JetBrains Mono 代码字体；CSS 变量集中在 `oj_frontend/src/index.css :root`。

## 2. 系统架构

```
浏览器 ──HTTP──> nginx(前端容器)
                   ├─ 静态资源(React 构建产物) + SPA 回退
                   └─ /api、/admin 反向代理 ──> backend(gunicorn, Django)
                                                   ├─ PostgreSQL（业务数据）
                                                   ├─ Redis（Celery broker/result）
                                                   └─ 提交判题入队 ──> worker(Celery)
                                                                          └─ 调用沙箱 go-judge(:5050) 编译/运行
```

**一次判题的流程**

1. 前端 `POST /api/submissions/`（标准题带 `code`，填空题带 `cloze_answers`）。
2. `SubmissionViewSet.create` 落库（status=pending），`judge_submission.delay(id)` 入队 Celery。
3. worker 执行 `apps/submissions/tasks.py::judge_submission`，按题型分流：
   - 标准题 → `apps/judge/runner.py::run_judge`；
   - 填空题 → `run_judge.judge_cloze`（评测机模式组装代码后复用 `run_judge`；文本模式逐空比对）。
4. `run_judge` 编译 → 逐测试点运行（沙箱）→ 比对 → 计分（含捆绑子任务）→ 回写 `Submission` 与 `SubmissionTestResult`。
5. 前端轮询 `GET /api/submissions/{id}/` 直到状态终态。

## 4. 数据模型

> 字段名以代码为准（`apps/<app>/models.py`），下表列主要字段与枚举。

### accounts.User（继承 AbstractUser）
| 字段 | 说明 |
|---|---|
| username | 登录名，**支持中文** |
| email | 可空、唯一（管理员建号可不填） |
| real_name | 真实姓名（竞赛/班级排名显示用） |
| role | `user` 学生 / `teacher` 教师 / `admin` 管理员 |
| email_verified | 是否已验证（管理员建号默认 True） |
| granted_permissions | JSON，额外授予的权限 key 列表 |
| accepted_count / submission_count | 解题数（去重）/ 总提交数 |
| avatar / bio / default_language / publicize_contest_code | 资料与偏好 |

属性/方法：`is_teacher`、`is_admin`、`effective_permissions`（角色基线∪额外授予，管理员=全部）、`has_perm_key(key)`、`display_name`（real_name 优先）。

### problems.Problem
| 字段 | 说明 |
|---|---|
| display_id | 题号（唯一，URL 用） |
| title / description / input_description / output_description / hint / source | 文本（Markdown） |
| difficulty | 洛谷分级（见枚举） |
| samples | JSON `[{input,output,note}]` |
| time_limit(ms) / memory_limit(MB) | 限制 |
| compare_mode | `default` 忽略行末空格 / `strict` 逐字节 / `float` 浮点误差；`float_precision` |
| spj_enabled / spj_language / spj_code | Special Judge |
| allowed_languages | JSON，允许语言（默认 `["python3","cpp"]`） |
| visibility | `public` / `hidden`(草稿) / `contest` / `class` |
| **problem_type** | `standard` / `cloze`(程序填空) |
| **cloze_template** | 填空模板，用 `__1__ __2__` 挖空 |
| **cloze_language** | 填空模板语言 |
| **cloze_use_judge** | True=评测机运行；False=文本比对 |
| **cloze_answers** | JSON，文本模式参考答案 `{"1":["float"],...}`（仅管理者可读） |
| tags / created_by / total_submit / accepted_count / created_at / updated_at | 其余 |

难度枚举（`Difficulty`，由低到高）：`unrated`暂无评定、`entry`入门、`pop_minus`普及−、`pop`普及/提高−、`pop_plus`普及+/提高、`imp_plus`提高+/省选−、`provincial`省选/NOI−、`noi`NOI/NOI+/CTSC。

### problems.TestCase
`problem, index, input_file(.in), output_file(.out), score, group, is_sample`
- `group=0` 独立计分；`group>0` 同组为**捆绑子任务**（整组全过才得该组分）。
- `is_sample=True` 为样例/可见（判题结果会回显其输入输出）。

### submissions.Submission
`user, problem, contest(可空), language, code, cloze_answers(JSON), status, score, time_used(ms), memory_used(KB), compile_error, first_failed_index, is_public, created_at`

评测状态枚举（`Verdict`）：`pending`等待、`judging`判题中、`accepted`通过、`wa`答案错误、`tle`超时、`mle`内存超限、`re`运行错误、`ce`编译错误、`pe`格式错误、`ole`输出超限、`se`系统错误。

### submissions.SubmissionTestResult
`submission, index, status, group, time_used, memory_used, score, input_preview, expected_output, actual_output`（后三者仅样例点有值）。

### contests.Contest / ContestProblem / ContestParticipant
- Contest：`title, description, start_time, end_time, rule_type(acm/oi), visibility(public/private/class), password, classroom(可空), penalty_minutes, freeze_minutes(封榜), created_by`
- ContestProblem：`contest, problem, label(A/B/C), score`
- ContestParticipant：`contest, user, registered_at`

### classes.ClassRoom / ClassMember / Assignment
- ClassRoom：`name, description, teacher, invite_code(邀请码), created_at`
- ClassMember：`classroom, user, role(student/assistant), joined_at`
- Assignment：`classroom, title, problems(M2M), assigned_at, due_at`

### discussions.Discussion / DiscussionReply
- Discussion：`problem(可空,版块), author, title, content, category, audit_status(published/hidden), reply_count`
- DiscussionReply：`discussion, author, parent(自引用,楼中楼), content`

### solutions.Solution
`problem, author, title, content(Markdown), language, audit_status(pending/published/rejected), reviewer, reject_reason, published_at`

### notifications.Notification / Announcement
- Notification：`recipient, type, title, content, link, is_read, created_at`
- Announcement：`title, content, created_by, created_at`

## 5. 权限与账号体系

### 角色与细粒度权限
- 角色：`user` / `teacher` / `admin`。管理员隐含拥有全部权限并可登录管理后台。
- 细粒度权限（可由管理员逐项开关，定义见 `apps/accounts/perms.py::PERMISSION_CATALOG`）：
  - `create_problem`：出题（创建并管理自己的题目）
  - `edit_any_problem`：管理所有题目
  - `review_solution`：审核题解
- 角色基线（`ROLE_BASE_PERMS`）：教师默认拥有 `create_problem`；管理员=全部；学生=空。
- 有效权限 = 角色基线 ∪ 额外授予。前端据 `user.effective_permissions` 决定头像菜单入口（出题管理 / 题解审核 / 管理后台）。

### 账号管理（仅管理员，已**关闭自助注册与改密**）
- 管理后台「用户管理」(`/admin`)：搜索、改角色、开关权限、启停账号、**新建账号 / 批量建号 / 重置密码**。
- 后端入口：`AdminUserViewSet`（`/api/auth/admin/users/`，含 `batch`、`set-password` action）。
- 登录仍为用户名+密码；管理员建号默认 `email_verified=True` 即时可用。

## 7. 判题机制（`apps/judge/`）

### 沙箱
- `JUDGE_SANDBOX=local`（默认，开发）→ `sandbox.LocalSandbox`，用 `os.fork`/`resource`/`setrlimit` 限制（仅 Linux）。
- `JUDGE_SANDBOX=go-judge`（生产）→ `gojudge.GoJudgeSandbox`，HTTP 调用 `GO_JUDGE_URL`（默认 `:5050`）。
- 统一接口 `BaseSandbox.compile / run`，由 `sandbox.get_sandbox()` 按环境变量选择。

### 语言（`languages.py`）
当前支持 `python3`、`cpp`（已移除 Java）。每种语言定义 `source_name / compile_cmd / run_cmd / use_address_space_limit / time_multiplier`。

### 比对（`comparator.py`）
`default`（忽略行末空格/末尾换行）、`strict`（逐字节）、`float`（按 `float_precision` 容差）；SPJ 启用时改由出题人脚本判定。

### 计分与捆绑子任务（重点）
`run_judge` 编译后逐测试点运行：
- 测试点按 `group` 分组。`group=0` 独立计分（通过即得该点分）。
- `group>0` 为捆绑子任务：**该组全部通过才得该组总分，任一失败该组 0 分**。
- 若题目含任意 `group>0`（捆绑题），会**跑完所有测试点**以判定各子任务；非捆绑题仍「首错即停」省时。
- 总状态：全部通过=AC，否则=第一个未通过点的状态；`submission.score` 为最终得分。

### 程序填空题（`judge_cloze`）
- 评测机模式（`cloze_use_judge=True`）：用学生 `cloze_answers` 把 `cloze_template` 里的 `__N__` 替换成完整代码，设好 `language=cloze_language` 后复用 `run_judge`（需配测试点）。
- 文本比对模式：逐空与 `cloze_answers` 参考答案比对，**忽略全部空白**（`a+b` 与 `a + b` 等价），每空可多个可接受写法；按答对比例计分，逐空写入 `SubmissionTestResult`。

### 在线运行（`runner.run_custom`）
编译并对自定义 stdin 运行一次，返回 `{status, compile_error, stdout, stderr, time_ms, memory_kb, exit_code}`，不评测不入库。对应接口 `POST /api/submissions/run/`。

## 8. 前端结构

- **路由**：`App.jsx` 集中定义。`/` 首页、`/problems`、`/problems/:displayId`（求解）、`/contests`、`/contests/:id`、`/discussions`、`/classes`、`/classes/:id`、`/profile`、`/settings`、`/login`、`/manage/problems`、`/manage/problems/:displayId/edit`、`/admin`、`/admin/solutions`。
- **API 层**：`api/client.js` 是 axios 实例（自动注入 JWT、401 刷新、统一错误）；`api/index.js` 封装所有接口并导出 `errMsg(e, fallback)`。新增接口都加在这里。
- **登录态**：`auth/AuthContext.jsx` 暴露 `useAuth() → {user, login, logout}`；`user` 含 `effective_permissions / is_admin / is_teacher / real_name` 等。
- **权限判断**：`utils/perm.js` 的 `hasPerm(user,key)`、`canManageProblems(user)`；菜单与页面入口据此显隐。
- **求解页**：`ProblemSolvePage` 按 `problem.problem_type` 选择 `EditorPanel`（标准题，Monaco + 运行/提交）或 `ClozePanel`（填空题，模板行内空位 + 编号输入）。结果统一用 `ResultPanel`（含子任务汇总）。
- **样式**：尽量复用 `index.css` 里的 CSS 变量与既有类（`.card`、`.page-title`、`.mono` 等）。

## 11. Docker 部署

```bash
# 首次：复制 .env.example -> .env，按需改 HTTP_PORT、SECRET、数据库密码等
docker compose up -d --build
# 访问 http://localhost:8080，登录 admin / admin12345
```
服务：`postgres`、`redis`、`go-judge`(privileged)、`backend`(gunicorn，启动时迁移+collectstatic+可选 seed)、`worker`(celery)、`frontend`(nginx)。卷：`pg_data / media / app_data / staticfiles`。

- 升级（保留数据）：覆盖代码后 `docker compose up -d --build`，入口脚本自动跑新迁移。
- 清库重建：`docker compose down -v && docker compose up -d --build`。
- 国内拉取慢：参考 `deploy/daemon.json.example` 配 Docker Desktop registry-mirrors；构建源用 `.env` 里的 `APT_MIRROR/PIP_INDEX_URL/NPM_REGISTRY/GH_PROXY`。
- 测试数据安全：`nginx.conf` 对 `/media` 一律 404，避免测试点（即答案）被直接下载。

## 12. 二次开发指引（常见改法）

**新增一门评测语言**
1. 后端 `apps/judge/languages.py` 加一项（`source_name/compile_cmd/run_cmd/...`）。
2. go-judge 镜像里安装对应编译器/运行时（`deploy/gojudge.Dockerfile`）。
3. 前端 `utils/templates.js` 的 `LANGUAGES` 与 `TEMPLATES` 加该语言（含 Monaco 语言名）。

**新增一种细粒度权限**
1. `apps/accounts/perms.py::PERMISSION_CATALOG` 加 `(key, 中文名)`；需要的话在 `ROLE_BASE_PERMS` 给某角色基线。
2. 后端用 `user.has_perm_key("key")` 或在 `apps/permissions.py` 加权限类来卡接口。
3. 前端 `utils/perm.js` 加判断；在 `AppLayout` 菜单/页面入口据 `hasPerm` 显隐。（目录通过 `/api/auth/admin/permissions/` 下发，后台开关自动出现。）

**新增一个后端接口**
- 资源型：在对应 app 的 `ViewSet` 加方法或新建 ViewSet 注册到该 app 的 `urls.py` router。
- 动作型：在 ViewSet 内用 `@action(detail=.., methods=[..], url_path="..")`。
- 别忘了在前端 `api/index.js` 加封装。

**新增一个前端页面**
1. `pages/` 下新建组件；2. `App.jsx` 加路由；3. 需要导航的在 `AppLayout` 加入口；4. 接口走 `api/index.js`。

**改判题/计分逻辑**
- 集中在 `apps/judge/runner.py`。捆绑计分在 `run_judge` 的「评分」段；填空在 `judge_cloze`；比对规则在 `comparator.py`；任务分流在 `apps/submissions/tasks.py`。

**新增一种题型**
- 仿照填空题：`Problem.problem_type` 加枚举值 + 相关字段（迁移）；`tasks.judge_submission` 分流到新判题函数；前端求解页 `ProblemSolvePage` 按 `problem_type` 选作答面板；详情序列化注意**不要泄露答案**给学生（参考 `cloze_answers_view` 的按权限返回）。

**数据迁移注意**
- 改模型后务必 `python manage.py makemigrations <app>` 并提交迁移文件；改枚举值且已有数据时写 `RunPython` 数据迁移（可参考 `problems/migrations/0003_remap_difficulty.py`）。

## 13. 已知限制与注意事项

- LocalSandbox 仅适合 Linux/开发；生产判题用 go-judge。
- 个人主页「提交通过率」基于已拉取的提交集合计算（未截断即全量；提交极多时为近 N 条窗口占比）。
- 竞赛排行榜为内存聚合，适合 ≤150 人规模；更大规模需改增量统计。
- 填空题文本比对「忽略全部空白」可容忍 `a+b`/`a + b`，但对「需要空格分隔的写法」（如 `return a`）请在参考答案里用 `|` 提供多个可接受写法，或改用「评测机运行」模式。
- 绝不可在面向学生的序列化里返回 `cloze_answers`/测试点答案。
- 演示管理员 `admin / admin12345` 仅供初始化，生产请立即改密或新建管理员后停用。
